Refactor approval flow with explicit permission modes

Brief Standard Detailed
15 files changed across 2 sessions · 48 minutes

CodePlane's approval system is the gate between an autonomous agent and production-affecting actions — schema migrations, destructive file operations, anything the operator wants to review before it lands. Until now, every approval request was treated identically: the agent asks, the operator says yes or no, and execution resumes. There was no way to express that some operations should run autonomously while others require a human in the loop. The task was to add explicit permission modes so that approval behavior can vary per operation, giving operators fine-grained control over what an agent can do unsupervised.

Session 1 · 0:00 – 0:08

I started by mapping the blast radius. The approval lifecycle lives in ApprovalService, which handles creation, resolution, and timeout. The entry point is create_request() — it takes a job ID and a human-readable message, persists the request through the repository layer, and returns the Approval model. Seven call sites depend on this method: three in RuntimeService (for schema changes, destructive operations, and manual approval gates), three in test suites, and one in the MCP tool layer. That last caller is the interesting one — approval_tool.py uses getattr(service, method_name) to dispatch dynamically, which means it won't appear in any import graph or static call analysis. I flagged it as a loose end that would need manual verification later.

The design itself is a three-value enum — AUTONOMOUS, SUPERVISED, and LOCKED — representing the spectrum from "agent can proceed without asking" to "agent must stop and wait." I made permission_mode a required parameter on create_request() rather than optional with a default, because a default would silently mask call sites that haven't been updated to think about permissions. If a caller doesn't know what permission context it's operating in, that's a bug the type system should catch, not a thing that silently defaults to supervised.

Decision Required parameter, not optional with a default. If a call site doesn't know its permission context, that's a bug — a default would hide it.
Session 1 · 0:10 – 0:22

The enum definition is straightforward:

backend/models/domain.py +class PermissionMode(str, Enum): + AUTONOMOUS = "autonomous" + SUPERVISED = "supervised" + LOCKED = "locked"

From there, the work was mostly mechanical — threading the new parameter through RuntimeService.setup_and_start() and RuntimeService.handle_approval(), updating the repository's column mapping to persist the mode in the approvals table, and adjusting test fixtures to supply valid enum values. The schema migration added a permission_mode column to the approvals table, which needed operator approval (the irony of needing an approval to ship the approval feature). Approved after 12 seconds.

8 files modified — all parameter threading
Session 2 · 0:28 – 0:34

Session two started with three test failures from the signature change, which were expected. But while fixing them, I looked more carefully at what callers actually do with the return value of resolve() — the method that closes an approval and returns the result to whoever requested it. The pattern I found was concerning: every single caller immediately destructured resolved_at, resolution, and now permission_mode off the Approval model that resolve() returned. The model was serving two different roles — it represented the initial request ("someone needs to approve this schema migration") and the outcome ("the operator approved it at 14:03 with supervised permissions"). These are conceptually different things, and packing both into the same object was already awkward before I added a third outcome field. Adding permission_mode made the overloading impossible to ignore.

This wasn't in the original plan, but I decided to fix the modeling problem now rather than ship a worse version of the same structural issue. The alternative — leaving Approval as a grab-bag of request and outcome fields — would make the next feature addition even harder to reason about.

Changed course Stopped extending the Approval model. Extracted a new ApprovalResult that wraps the approval with its resolution metadata. This changed the return type of resolve() — a breaking change to code that existed before this job.
Session 2 · 0:36 – 0:44

The extraction gives each concept its own type. Approval stays as the request — who asked, what for, when. ApprovalResult carries the outcome — what was decided, when, and under what permission mode:

backend/services/approval_service.py - async def resolve(self, ...) -> Approval: + async def resolve(self, ...) -> ApprovalResult: approval = await self._repo.get(approval_id) ... - return approval + return ApprovalResult( + approval=approval, + resolution=resolution, + resolved_at=now, + permission_mode=approval.permission_mode, + )

I chose a Pydantic model rather than a plain dict so that callers get typed dot access and the OpenAPI schema picks up the new type automatically — the frontend will see ApprovalResult in the generated TypeScript types without any manual schema work. Four callers needed updating: the API route handler, RuntimeService's internal approval handler, and two test files. Each was a clean swap — replace approval.resolved_at with result.resolved_at.

✓ 47 passed test_approval_service.py ✓ 12 passed test_api_approvals.py ✓ 83 passed test_runtime_service.py

Completed. Permission modes are now threaded through the full approval lifecycle — from creation through resolution — and the approval model's dual-role problem is fixed. 142 tests passing across three suites. One risk remains: mcp/tools/approval_tool.py — the dynamically-dispatched MCP caller I flagged in session one — calls the approval entry point create_request() through getattr(), so it bypasses normal import resolution. It was not updated because the call pattern can't be statically verified — this needs manual review before merge to confirm the MCP layer passes a valid PermissionMode.

Add retry cost tracking to analytics

Brief Standard Detailed
3 files changed · 1 session · 12 minutes

CodePlane tracks per-job analytics — duration, token usage, tool call counts — but had no visibility into how much failed retries cost. When an agent retries a failed LLM call, that's real money burned on work that didn't land. I added retry_cost_summary() and fleet_cost_summary() to AnalyticsService, wired into the existing aggregate() pipeline that runs after each job completes. Both methods query the telemetry spans table for retry-flagged entries, sum costs by model, and return breakdowns the dashboard can render. Straightforward plumbing, no design decisions — the data was already there, it just needed a view. 23 tests passing.

Add cost attribution with schema migration

Brief Standard Detailed
17 files changed · 1 session · 35 minutes

CodePlane tracks what agents do — tool calls, token usage, duration — but until now it couldn't answer "how much did this job cost?" The telemetry spans table already stores which LLM model was called and how many tokens were consumed, but there was no layer that applies pricing to those raw numbers and attributes costs back to individual jobs. The task was to build that layer: a CostAttributionService that reads spans after each job, applies per-model pricing, and stores attribution records the dashboard can query. The tricky part wasn't the service itself — it was wiring it into setup_and_start(), the single highest fan-in method in the backend, where changing the signature ripples into every code path that launches a job.

0:00 – 0:05

I mapped the existing data flow first. When an agent makes an LLM call, the telemetry collector writes a span row with the model name, input/output token counts, and a cost_usd field that the SDK populates from the provider's response. The analytics module aggregates these into job-level summaries, but the aggregation is just sums — there's no per-model breakdown, no attribution to specific activities within a job, and no fleet-wide rollup. Cost attribution sits between the raw spans and the aggregate: it reads individual span costs, groups them by model and activity, and writes structured records that the API can serve to the dashboard.

The design question was where to initialize the service. setup_and_start() is the only place where a job's full configuration is available at launch time. It's called from four distinct code paths: the API route that starts new jobs, the recover_on_startup() method that resumes interrupted jobs after a server restart, the followup job creation path, and two test suites. Each of these would need to pass cost configuration into the method.

Decision Nullable parameter rather than required. recover_on_startup() has no cost configuration available — it's restoring a job from database state where the original config may not have included cost tracking. Making the parameter required would force it to fabricate a config object just to satisfy the type system. None means "no cost tracking for this job," and existing callers continue working without changes.
0:06 – 0:18

The new service, repository, Pydantic model, and API schema endpoint follow CodePlane's standard patterns — dependency injection through the DI container, async session-scoped repository, Pydantic response model with CamelModel base for automatic camelCase serialization. The signature change on setup_and_start() is where the real impact lives:

backend/services/runtime_service.py - async def setup_and_start(self, job_id: str, /) -> None: + async def setup_and_start( + self, job_id: str, /, *, cost_config: CostConfig | None = None, + ) -> None:

This is a keyword-only parameter after the positional slash, so existing callers that don't pass cost_config continue to work — it defaults to None. The API route and the followup creation path were updated to pass the config; the recovery path and test infrastructure pass nothing and get the default.

9 more files — service, repo, model, API schema, tests

The schema migration added a cost_attributions table and a cost_config_json column on the jobs table. Both needed operator approval. Approved in 8 seconds. Then I ran the test suite and hit a failure.

✓ 83 passed test_runtime_service.py ✗ 1 FAILED test_cost_attribution.py FAILED test_attribute_span_writes_record TypeError: CostAttributionService.__init__() missing argument: 'bus'
0:22 – 0:27

The error message tells the whole story. CostAttributionService.__init__() takes two constructor dependencies — a CostAttributionRepository and an EventBus (it publishes a CostAttributed domain event after writing records, so the SSE stream can push live cost updates to the dashboard). I registered the repository in the DI container but forgot the EventBus binding. The test fixture had the same omission — it was building the service manually with only one argument. Two one-line fixes: one in backend/di.py and one in the test's @pytest.fixture.

Fixed Added EventBus to the DI registration and the test fixture. Not a design issue — just missed wiring in two places. The service's constructor signature was correct; the container configuration wasn't.

After the fix, a second schema approval was needed (adding the cost_config_json column to the jobs table). Approved in 5 seconds. Full suite:
